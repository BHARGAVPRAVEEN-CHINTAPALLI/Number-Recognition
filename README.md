# ✍️ Handwritten Digit Recognition

A Streamlit web app that recognizes hand-drawn digits (0–9) using a Convolutional Neural Network (CNN) trained on the MNIST dataset. Draw a digit on an in-browser canvas and get a real-time prediction with confidence scores.

## Overview

This project has two halves:

1. **Training pipeline** (`run.py`) — trains a CNN on the MNIST handwritten-digit dataset and saves the trained model.
2. **Inference app** (`app.py`) — a Streamlit UI with a drawable canvas. The user draws a digit, the image is preprocessed to match MNIST's format, and the trained model predicts which digit (0–9) it is, along with a confidence score and full probability breakdown.

## Architecture

```
┌─────────────────┐      ┌──────────────────┐      ┌───────────────────┐
│   MNIST Dataset  │ ───▶ │   run.py (train)  │ ───▶ │   mnist.keras      │
│  (60k train imgs) │      │  CNN: Conv→Conv→  │      │  (saved model)     │
└─────────────────┘      │  Pool→Dense→Dense │      └─────────┬─────────┘
                           └──────────────────┘                │
                                                                ▼
┌─────────────────┐      ┌──────────────────┐      ┌───────────────────┐
│  User draws on   │ ───▶ │  Preprocessing    │ ───▶ │   app.py loads      │
│  canvas (400x400) │      │  crop/resize/     │      │   model, predicts,  │
│                   │      │  center to 28x28  │      │   shows result      │
└─────────────────┘      └──────────────────┘      └───────────────────┘
```

**Model architecture** (defined in `run.py`):

| Layer | Details |
|---|---|
| Conv2D | 32 filters, 3×3 kernel, ReLU |
| Conv2D | 64 filters, 3×3 kernel, ReLU |
| MaxPooling2D | 2×2 pool |
| Dropout | 0.25 |
| Flatten | — |
| Dense | 128 units, ReLU |
| Dropout | 0.3 |
| Dense | 10 units, Softmax (output) |

## Pipeline Details

### 1. Data loading & preprocessing (training side — `run.py`)

- Loads MNIST via `tensorflow.keras.datasets.mnist` (60,000 train / 10,000 test images, 28×28 grayscale).
- Reshapes images to `(28, 28, 1)` and normalizes pixel values to `[0, 1]`.
- One-hot encodes the 10 digit labels.

### 2. Model training (`run.py`)

- Compiles with `categorical_crossentropy` loss and the `adam` optimizer.
- Trains for 15 epochs, batch size 128, with validation against the test set.
- Saves the trained model as **`mnist.keras`** (native Keras 3 format — not the legacy `.h5` format; see [Troubleshooting](#troubleshooting) for why this matters).

### 3. Image preprocessing (inference side — `app.py`)

The canvas captures a 400×400 RGBA image that looks nothing like raw MNIST data, so it's transformed to match:

1. Convert to grayscale.
2. Invert colors (canvas is black-on-white; MNIST is white-on-black).
3. Crop to the drawn digit's bounding box (removes empty margins).
4. Resize the digit so its longest side is 20px, preserving aspect ratio.
5. Center the result on a blank 28×28 canvas (MNIST's exact format).
6. Normalize to `[0, 1]` and reshape to `(1, 28, 28, 1)` for the model.

### 4. Prediction (`app.py`)

- `model.predict()` returns a 10-element probability vector.
- The digit with the highest probability is shown as the prediction, alongside its confidence percentage and a full per-digit probability breakdown.

## Running Locally

### 1. Train the model (skip if `mnist.keras` already exists)

```bash
python run.py
```

This downloads MNIST automatically on first run and trains for a few minutes on CPU. It prints training/validation accuracy per epoch and saves `mnist.keras` in the project root when done.

### 2. Launch the app

```bash
streamlit run app.py
```

Open the printed local URL (typically `http://localhost:8501`), draw a digit, and click **Predict**.

## Deploying to Streamlit Cloud

1. Push `app.py`, `run.py`, `mnist.keras`, and `requirements.txt` to a GitHub repo.
2. On [share.streamlit.io](https://share.streamlit.io), click **New app**, select your repo/branch and `app.py` as the entry point.
3. In **Advanced settings**, set the Python version explicitly (this project is built and tested against **Python 3.11**).
4. Deploy. Streamlit Cloud installs `requirements.txt` and starts the app automatically.
5. Future pushes to the connected branch auto-redeploy.

## Tech Stack & Versions

| Package | Version | Why this version |
|---|---|---|
| Python | 3.11 | TensorFlow does not yet support the newest Python releases; 3.11 is a stable, well-supported target. |
| streamlit | 1.49.0 | Minimum version required by `streamlit-drawable-canvas-fix`. |
| tensorflow | 2.17.0 | Mature, widely-deployed release. Newer TF releases (2.2x) have shipped with ABI mismatches against the Abseil/NumPy binaries available in some hosted environments. |
| pillow | 10.4.0 | Image preprocessing (crop/resize/center). |
| streamlit-drawable-canvas-fix | 0.9.8 | The original `streamlit-drawable-canvas` package is unmaintained/archived and breaks on modern Streamlit versions. This is an actively-maintained drop-in replacement exposing the identical `st_canvas` API. |

`numpy` is intentionally **not pinned** — it's left to `pip`/`uv` to resolve automatically based on what TensorFlow itself declares as compatible, to avoid binary ABI mismatches (see Troubleshooting).

## Troubleshooting

This project went through several real deployment issues worth documenting for future reference:

**`ModuleNotFoundError: No module named 'tensorflow'` (local)**
Dependencies not installed in the active environment. Run `pip install -r requirements.txt` in the same environment used to run `streamlit run app.py`.

**Build fails resolving `tensorflow==X` against Python 3.14**
TensorFlow doesn't ship wheels for every Python version immediately. Pin the Python version explicitly at deploy time (Streamlit Cloud: Advanced settings → Python version) rather than relying on `runtime.txt`, which Streamlit Cloud no longer honors.

**`TypeError: ... Unrecognized keyword arguments: ['batch_shape', 'optional']` on model load**
The model was saved with one Keras version (e.g. standalone Keras 3) and loaded with a different, incompatible one (e.g. TF-bundled Keras 2) — these are different libraries with different serialization formats despite the shared name. Fix: train and serve with the exact same `tensorflow` version, and use the native `.keras` save format (`model.save("mnist.keras")`) instead of legacy `.h5`.

**`OSError: No file or directory found at mnist.keras`**
The trained model file wasn't committed/pushed to the repo. Generated artifacts like `mnist.keras` must be added to git explicitly — they aren't created by `git push` itself.

**`streamlit-drawable-canvas` import or rendering errors on modern Streamlit**
The original package is archived/unmaintained and incompatible with current Streamlit internals. Use `streamlit-drawable-canvas-fix` instead — same `import streamlit_drawable_canvas` API, actively maintained.

**`NotFoundError: ... undefined symbol ...` / NumPy 1.x vs 2.x ABI warnings on TensorFlow import**
A binary ABI mismatch between TensorFlow's compiled kernels and the NumPy/Abseil versions resolved alongside it — most common with very recently released TensorFlow versions. Fix: pin TensorFlow to a mature, widely-deployed release (this project uses 2.17.0) and avoid separately pinning `numpy` — let pip resolve it from TensorFlow's own declared constraints.

**General rule:** train and deploy with the *exact same* `tensorflow` version. Mismatches between training and serving environments are the source of nearly every model-loading bug.
